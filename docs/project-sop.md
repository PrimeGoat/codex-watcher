# codex-watcher SOP

## Role

This document is the canonical operational doctrine for ingest, initialization, realization tasking, continuation, and startup awareness routing.

## Startup Awareness Chain

At session start, load awareness in this order:

1. `AGENTS.md`
2. `AI.md`
3. Atlas startup index knowledge entry
4. canonical `project_state`
5. the specific canonical repo docs and Atlas knowledge needed by the active task

`AI.md` stays brief and points to this SOP, the Atlas startup index, and `project_state`.

## Canonical Surface Routing

When ingesting or updating project information:

- substantive project and runtime truth goes into canonical repo docs
- compact durable extracts go into Atlas knowledge and reference repo doc path/filename when derived from fuller docs
- compact current truth goes into the canonical `project_state`
- realization-driving work goes into Atlas tasks

## Initialization Doctrine

For the project's first real session, the required order is:

1. initialize the new project's own management system
2. ingest the bootstrap source into the new project's canonical surfaces
3. create the required task set in the new project's own namespace/system
4. reflect settled decisions into the new project's own `project_state` and repo-local docs
5. continue from the resulting initialized state

## Realization Doctrine

The Atlas task graph is the work structure for this project.

Tasks should realize canonical artifacts by reference rather than duplicating canon in paraphrased form.

Dependencies should be direct only.

Sequential ordering should be used only when a task's purpose requires it.

All realization-relevant tasks should exist up front.

Continuation is represented by the task `Determine how to proceed`.

## Dependency Rule

The required design tasks are prerequisite work for the initial export-ready architecture state.

The project should not proceed to full implementation as if those design surfaces were already settled.

## Source Retirement

The bootstrap ingest source may be moved to `tmp/` after its contents have been correctly projected into canon and tasking, and after that projection has been verified.
